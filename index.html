Clean Architecture 簡潔架構

前言

為甚麼需要軟體架構？
Uncle Bob 在書中提到了這個概念，使用軟體架構是為了最小化建置和維護系統時所需要的成本。

“The goal of software architecture is to minimize the human resources required to build and maintain the required system.” ― Robert C. Martin, Clean Architecture

在執行專案時，我們都會希望開發的時程越短越好，但如果在開發時沒有去運用架構，僅僅只是完成需求上的功能就好，就有可能將這些忽略的開發成本無形地轉嫁到後續的維運成本。
舉例來說，一個已經上線的專案突然要改變 UI，如果沒有設計好架構，可能就要連動到商業邏輯的修改，導致你在維運階段就需要花更多時間去測試和滿足需求。
因此我們會需要了解簡潔架構，簡潔架構會透過三個原則，分別是相依性原則、分層原則和跨層原則來達到關注點分離，也就是說系統核心不會知道外部所使用資料庫、框架和 UI 是甚麼，只需要關注本身的商業邏輯。後續即便有需求變更，也可以更快速聚焦和修改，維運成本也隨之降低。

要特別提到的是這邊提到的三大原則並沒有在 Clean Architecture 書中所定義，而是先前我在上Teddy的課程中，他對於Clean Architecture全書做的總結，因我覺得這樣能更容易理解簡潔架構，所以在這篇文章中會使用三大原則來說明簡潔架構

來源請參考: https://teddy-chen-tw.blogspot.com/2020/08/clean-architecture.html

使用簡潔架構的優勢

  • 獨立框架

    將框架當作工具來使用，而工具是可以被替換的，也就是系統內層不會依賴於框架

  • 獨立 UI

    UI的變動不會影響到商業邏輯，因 UI 時常會因滿足客戶需求而進行變動

  • 獨立 Database:

    將資料庫也作為工具使用，舉例來說，原先使用 MySQL，但後續想要改使用 MongoDB 也能直接替換卻不會影響到商業邏輯。套用到其他中間件也是，像是 Message Queue，一樣也能將 kafka 或 RabbitMQ 當作工具來使用

  • 獨立外部服務:

    將商業邏輯與外部系統切開

  • 可測試:

    商業邏輯可以在沒有 UI、資料庫和其他中間件等等的情況下進行測試

簡潔架構三原則

分層原則

那在簡潔架構中，主要我們會分成4層，而每一層都有其負責的用途及功能：

  • Entitiy

    商業邏輯層，Entity 可以是一個物件，也可以是一組資料結構，Entity 不會受到外部的影響(像是DB或UI)，關注的重點只有他自己。

  • UseCase

    使用案例層，系統中所有的使用案例都會封裝在這一層，控制 entities 資料的流動，並在這層透過依賴反轉去操控外部系統 (e.g. DB) 去滿足使用案例。舉例來說，建立、修改及刪除會員都可以個別視為一個 UseCase。

  • Interface Adapter

    接口轉換層，負責資料的接收和資料格式的轉換。舉例來說，當會員資料透過 RESTful API 傳送到系統，就會由這一層接收，並且做轉換再傳入到 UseCase 層。

  • Framework & Driver

    負責框架、DB 和外部服務的相關設定及實作

相依性原則

從上面的圖可以了解到簡潔架構是由四個同心圓組成，而越往內層，層級就越高。最內層是處理商業邏輯的核心，而外層就偏向於處理流程上的機制，因此程式碼的依賴只能由外往內。簡單來說，內層的程式碼不能引入外層的變數、函式或類別等等，這就是相依性原則。

跨層原則

簡潔架構在做跨層資料的輸入或輸出時，都會使用基本的資料結構來傳遞，也就是將原先的物件再重新包裝程一個純資料結構的物件再傳送，這樣可以避免違反相依性原則。

舉例來說，當建立會員時，會先由 Adapter 層接收 Restful API 的請求，而當我們需要調用 UseCase 層去新增會員時，就會需要先建立一個純POJO在UseCase層讓Adapter進行轉換，藉此UseCase層就不會引用到外層的東西。

結論

在以往的開發過程中，如果我們使用MVC架構，因為沒有相依性原則和跨層原則的規範，就會是直接建立實作類別。而在缺少了介面定義的情況下，單元測試就會相對難寫。另外在沒有單元測試的情況下，維護上就容易出現改Ａ壞Ｂ，增加維運的成本。所以我們可以透過簡潔架構來將系統進行分層，將商業邏輯和外部系統、框架做一個解耦，一方面是我們能更彈性的去抽換中間件，另一方面則是在測試上更容易可以去撰寫。

甚麼是 Scope

Scope 可以翻成 作用域，也就是指「變數如何被存取」，能用下面兩個問題來概括解釋。

Where do variables live?

Where can we access a certain variable?

3種型態的 Scope

  • Global Scope
    全域變數，除了 Function Scope 和 Block Scope 所定義的變數，都在此範圍

  • Function Scope
    在函式中宣告的變數，只能在函式中取用

  • Block Scope
    Block 指的是被大括號包起來的範圍，被宣告的變數只能在此範圍中存取

Lexical Scoping 和 Scope Chain

Lexical Scoping 可以翻成「語彙範疇」，也就是指函式和Block的擺放位置，
變數能不能被取用是由擺放位置來決定，而不是函式呼叫的順序。

至於 Scope Chain 則會依據 Lexical Scoping 來決定說，
變數能否向上尋找和取用(Variable Lookup)，
很重要的一點，外層Scope不能存取內層的Scope的變數，只會為單向取用。

甚麼是執行環境 execution context

所有JS的程式碼都會在執行環境中執行，
但只會有一個 Global exection context。
而每一個function則會在各自的exection context中執行。
因此執行環境可以分成兩種，

  • Global Execution context
  • Function Execution context

而每個execution context可分為建立階段和執行階段，

  • 建立階段: 參數和函式宣告式會被裝進execution context

    • varialbe environment
      • let, const and var declartion
      • functions
      • arguments object
    • scope chain
    • this keyword

  • 執行階段: 每一個execution context都會被放到call stack，stack會一直往上堆疊，最上面的會先被執行，執行完成後就會被拿掉。

Js Fundamental

發表於 2022-12-17 更新於 2023-12-16 分類於 Javascript

What is JavaScript

  • High level
    • JS 不需要考慮記憶體或資源配置
  • Garbage collected
    • JS 會自動清除記憶體
  • Interpreted or just-in-time compiled
    • 轉換成 Machine code 在 JS engine
  • Multi Paradigm
    • Procedural programming
    • Object oriented programming (OOP)
    • Functional programming (FP)
  • Prototype-based object oriented
  • First-class functions
    • functions 被視為一種 變數
  • Dynamic
    • 弱型別，變數可以在 runtime 改變型別
  • Single-threaded
  • Non-blocking event loop
    • concurrency model 讓 Js engine 掌控多個任務在一個時間
    • 因為 JS 是走 single thread
    • 使用 eventloop 讓某些長時間運作的任務，可以在背景執行

JS Engine

Program that executed JS code.
Every JS Engine have two kind of things, one is call stack, the other is heap.
Stack is execution context is out code get executed.
Heap where objects are stored in memory.

甚麼是 compilation, interpretation
compilation 是 source code 先轉化成 machine code ， 會有一個binary file，然後在透過任何方式執行
interpretation 是 source coee 每轉一行 machine code 就執行一次 (line by line)

Modern JS
Just-in-time (JIT) compilation
Source code也會直接轉化成 Machine code 但不會有檔案產生，並且會被馬上執行

當有程式碼進到 JS Engine 流程：

  • Parsing ( Abstract Syntax Tree )
  • Compilation (convert to machine code)
  • Execution (發生在 call stack)
  • Optimization (在 execution 期間)，增加效能加快速度
