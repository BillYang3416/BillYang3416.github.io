淺談微服務架構的非同步機制

發表於 2024-01-05 更新於 2024-01-06 分類於 Microservice

前言

本篇文章會先說明同步和非同步的差別，接著在解釋在微服務中使用非同步的優勢，
最後則簡單介紹非同步的通訊協定，像是 AMQP。

目錄

  • 甚麼是同步和非同步
  • 在微服務中使用非同步的優勢
  • Advanced Message Queuing Protocol (AMQP)

甚麼是同步和非同步

以往有接觸過 Web Service 的朋友，應該都對同步和非同步的概念並不陌生，
這邊我們舉一個例子來說明同步和非同步的差別，
假設在電商系統中，我們有兩個服務分別是訂單和庫存，
訂單服務接收到前端建立訂單的 HTTP 請求，當訂單成立時，我們需要去減少庫存的商品數量，
這時候訂單服務會再發送 HTTP 請求給庫存服務，庫存對資料庫操作後再回應給訂單服務，
訂單服務再回應給前端表示建立訂單成功，這個資料流程就會為同步。
透過下方的圖示可以看到訂單和庫存服務已經耦合在一起，並且前端接收到回應時間也會被拉長。

所以我們可以使用非同步的機制來將服務間的關係解耦並且減少回應時間，
透過 Message Broker 建立生產者和消費者的 Pattern，
生產者可以持續將訊息放入到 Message Broker，再由消費者透過訂閱的方式向 Message Broker 取得訊息在進行處理，這種資料流程就會為非同步。

在微服務中使用非同步的優勢

那為甚麼我們需要在微服務中使用非同步的機制呢?
其實從前面的例子可以了解到幾個優勢:

  • 解耦服務之間的依賴關係

    假設我們使用同步的方式處理資料，就會遇到當庫存服務改變 API 的 URL 時，連帶就會影響到訂單服務；使用非同步能減少服務間的直接依賴

  • 增加效能

    因服務間的資料流不會被同步所阻塞，所以回應時間就會與之降低，增加系統的效能

  • 容錯率變高

    因服務之間不會直接依賴，所以當庫存服務壞掉時並不會影響到訂單服務的流程，所造成的影響可能只是 Message Broker 的 Queue 會儲存大量的訊息

因為這些優勢，所以在微服務的架構中都會設計非同步的資料流和 Message Broker 的角色。

Advanced Message Queuing Protocol (AMQP)

在微服務的架構中，通常我們會使用 AMQP 作為非同步的通訊協定，
AMQP 也和 HTTP 一樣，屬於 OSI 中的 Applicaion Layer，
通常會被 Message-Oriented 的 Middleware 所使用，用來和服務做資料交換。
而使用 AMQP 的 Middleware 主要有兩個，分別是 RabbitMQ 和 ActiveMQ。

當然在微服務中也會使用其他的 Middleware 像是 Apache Kafka、Amazon SQS 和 Google Pub/Sub，這些都沒有支援 AMQP，但一樣能達到非同步的目的和取得它的優勢。

結論

透過這篇文章的說明，可以了解到非同步機制的設計在微服務中不可或缺，
對此有興趣的讀者可以在深入了解 AMQP 的原理和 middleware 的使用方法，
像是 RabbitMQ 的官方文件就相當完整，在參閱後就能簡單建立出具有非同步機制的服務。

GoRoutine

GoRoutine 基本上可以說是 Go Concurrency 的基礎之一，所以要學會使用和理解它的概念，
才能活用 Go 的 Concurrency Patteren，否則在開發上就會遇到無數的瓶頸和意想不到的 Bug。
這邊來說明三個 GoRoutine 的特點：

  • 輕量化

    相較於 Thread 所使用的記憶體和啟動時間，GoRoutine 所使用的資源比例都遠低於 Thread，可以說 GoRoutine 就是輕量化的 Thread

  • 抽象化

    當我們使用 GoRoutine 時不用考慮到 OS 層級，因為 Go Runtime 會幫你管理 GoRoutine 和 OS Thread 之間的協作、溝通

  • 簡單上手

    建立 GoRoutine 只需要在函式或方法前加上 go 的關鍵字，就能簡單地寫出 Concurrency 的程式碼

Unbuffered and Buffered Channel

前面說明了 Channel 可以讓 GoRoutine 之間交換資料和達成同步，
而 Channel 儲存資料的容量又可以分為 Unbuffered 和 Buffered。
Unbuffered 的例子就是我們前面所使用的程式碼 c := make(chan string)，
代表這個 Channel 沒有容量可以存放，當 sendHello GoRoutine 要執行傳遞資料時，
Main GoRoutine 必須準備接收，否則傳遞就會被 Blocked，發生阻塞的現象。

而在某些情況下，我們會希望降低阻塞現象的發生，像是在日誌系統中可以有生產者和消費者的角色，
生產者負責產生及傳遞日誌資料給 Channel，而消費者則負責從 Channel 接收日誌資料寫入到檔案，
因為日誌的資訊是非常重要的，所以我們需要生產者能持續產生，
因此不希望生產者發生阻塞現象，此時就可以使用 Buffer Channel，
代表這個 Channel 在未達到最大容量前都不會被 Blocked。

發表於 2023-12-23 更新於 2024-01-06 分類於 Golang

前言

相信很多人在學習 GO 以前，都知道 GO 在處理並發時相對其他程式語言具有優勢，
我們能使用 GoRoutine(輕量化的Thread) 搭配 Channel 來處理並發，而且撰寫上也相對簡單。
因此這篇文章首先會介紹 Concurrency 的概念，再來說明如何使用 WaitGroup 來處理並發，
並且透過 Mutex 和 Atomic 來處理並發產生的 Race Condition，
至於 Channel 則會在未來的文章會與大家做介紹。

目錄

  • Concurrency
  • WaitGroup
  • Race Condition
    • Mutex
    • Atomic

Concurrency

Concurrency is not parallelism.

這句話是 Rob Pike 在 Concurrency is not Parallelism 的影片中對 Concurrency 和 Parallelism 的解釋，非常建議讀者們可以看這部影片，有助於對 Concurrency 概念有更深入的理解。

接下來我們就來透過例子來了解什麼是 Parallelism 和 Concurrency，
這個例子是量販商的工作人員正在透過推車將電商貨品搬到貨車上。

Parallelism

假設到了過年期間，貨量突然暴增，店長覺得這樣出貨的速度太慢，
因此他決定多加一個出貨站、一位工作人員和一台貨車，
藉此平行化的處理同一件事讓出貨效能變好，這就是 Parallelism 。

Concurrency

那我們換一個想法，當我們不增加一個出貨站和一台貨車時，我們能怎麼做呢？
我們可以增加3位工作人員，讓他們各分其職，
有一位負責在出貨站出貨、而另一位則在貨車前面上貨，而另外兩位則負責推車。
基本上沒有工作人員在做重複的事情，而是有組織地讓每一個工作人員分工合作，
這就是 Concurrency。

Concurrency vs Parallelism

透過上述的例子可以看到 Parallelism 是通過加大資源在同一時間平行化地來執行任務，
而 Concurrency 則是有結構性、組織化的來管理和執行任務。
因此可以說 Concurrency 並不是 Parallelism，
而在 Go 中，我們使用 Goroutine 和 Channel 就是要透過 Concurrency 來設計你的任務，
藉此能負載高吞吐量的環境。

另外也說明了如何使用 WaitGroup、Mutex 和 Atomic 來實作 Go 的 Concurrency。

深入解析 RabbitMQ 的參數設定

發表於 2023-12-22 更新於 2024-01-06 分類於 RabbitMQ

前言

當我們在建立 RabbitMQ 的物件時，像是 Exchange、Queue 或 Binding，
我們可能會需要設一些參數上去，像是 message-ttl 或 max-length 等等。
而這些參數通常會需要在 runtime 期間可以被改動的，
所以我們會使用 Policy 來配置這些參數，這也是 RabbitMQ 官方所推薦的作法。
這篇文章會介紹這些參數的用途和如何使用 Policy 進行配置。

以 Queue 來說，如果建立時有綁定 message-ttl 就代表無法更改值，除非將 Queue 砍掉在重建。

目錄

  • Exchange
  • Queue
  • Binding
  • 閥值類參數的建議

Exchange

Exchange 的參數可以劃分為靜態參數和可選性參數，
而可選性參數都建議都使用 Policy 來配置，
因為如果建立時就配置可選性參數，就會需要刪除物件才能修改參數的設定。

靜態參數

名稱 用途 預設值
type Exchange 的類型，像是 direct、fanout、topic 和 header
durable broker 重新啟動後仍然存在
auto_delete 當沒有 Queue 與 Exchange 做 binding 時會自動刪除 false
internal Exchange 不能接收發送者的訊息，只能在 broker 內部用來交換訊息 false

可選性參數

名稱 用途 預設值
alternate-exchange 當訊息無法被轉到對應的Queue時，訊息可以轉到替代的 Exchange

Queue

至於 Queue 一樣也可以分為靜態參數和可選性參數，
而可選性參數都建議都使用 Policy 來配置，
因為如果建立時就配置可選性參數，就會需要刪除物件才能修改參數的設定。

靜態參數

名稱 用途 預設值
durable broker 重新啟動後仍然存在
auto_delete 當沒有 Binding 也沒有消費者連接時就會自動刪除 false

可選性參數

名稱 用途 預設值
x-message-ttl 訊息能在 Queue 裏存活的時間，如果時間超過，訊息就會被轉至到 DLQ 或刪除。另外如果訊息的狀態已經變成　unacknowledge 就不會計算 ttl
x-expires 如果 Queue 在時間內都沒有被使用到，則 Queue 會被自動刪除
x-max-length 訊息在 Queue 裏的最大存放數量，如果訊息的狀態已經變成　unacknowledge 則不在此列
x-max-length-bytes 訊息在 Queue 裏的最大存放空間，如果訊息的狀態已經變成　unacknowledge 則不在此列
x-overflow 當 x-max-length 或 x-max-length-bytes 的條件達成時，Queue 對接收的訊息所做出的行為，可以分為 drop-head (最舊的會被移除)、reject-publish (發送端會發送失敗) 和 reject-publish-dlx (發送失敗被轉移到 DLQ) drop-head
x-dead-letter-exchange 當訊息被 rejected 或過期時，要經由哪一個 Exchange 做轉移
x-dead-letter-routing-key 當訊息被 rejected 或過期時，用來轉移 DLQ 的鍵值
x-queue-mode 訊息要儲存在 Memory 還是 Disk，可以分為 default(memory) 和 lazy(disk) default
x-queue-type Queue 的類型，可以分為 classic 和 quorum classic

Policy 額外能設定的參數

名稱 用途 預設值
delivery-limit 當訊息沒有被 RabbitMQ 傳遞成功時，RabbiMQ 嘗試傳遞的最大次數，避免發生 Poison Message 的現象，也就是當消費者消費訊息失敗，訊息回到 Queue，RabbitMQ又在傳遞給消費者，如此重複下去
dead-letter-strategy 當訊息傳遞失敗或超過TTL，在傳遞給DLQ的策略，可以分為 at-most-once(訊息最多傳遞一次給DLQ) 和 at-least-once(訊息至少要傳遞給DLQ成功一次，必須搭配 overflow 的 reject-publish) at-most-once

Binding

Binding 是用來配置 Exchange 和 Queue 的關係，
這一塊通常不需要使用 Policy 來配置可選性參數，
只需要在建立的時候指定 Exchange、Queue 和 RoutingKey。

靜態參數

名稱 用途 預設值
Exchange Exchange 的名稱
Queue Queue 的名稱
Routing Key 路由的鍵值

設置閥值類參數的建議

前面我們提到了 Queue 的可選性參數，其中有三個參數分別是 x-message-ttl、x-max-length 和 x-max-length-bytes，
這三個可以歸類成閥值類的參數，代表我們需要依據系統來動態調整這些參數。
這邊提供一些調整參數時的準則當作參考:

  • x-message-ttl
    • 消費者行為: 如果消費者消費比較慢，則要將 ttl 設長一點
  • x-max-length
    • 吞吐量大小: 計算 publising rate 和 consuming rate，如果生產者產製訊息比消費者更快，就需要更大的 x-max-length
  • x-max-length-bytes
    • 訊息大小: 如果每一筆訊息的容量都偏大，就可以將 x-max-length-bytes 調高，以便負荷更多的訊息

結論

在這篇文章我們詳細介紹了關於 Exchange、Queue 和 Binding 的參數，
並且還區分了靜態參數和可選性參數。
另外值得一提的是，使用 Policy 來配置可選性參數相當重要，
因為在系統運行時，我們不希望要刪除物件來修改參數，
一旦刪除了物件，不只會影響到服務的運行，還要重新配置 Exchange 和 Queue 的 Binding，一不小心就很容易出錯。
最後則提供了一些配置閥值類參數的建議。

Clean Architecture 簡潔架構

發表於 2023-12-12 更新於 2024-01-06 分類於 Golang ， Clean Architecture

目錄

  • 前言
  • 使用簡潔架構的優勢
  • 簡潔架構三原則
  • 結論
  • 參考資料

前言

為甚麼需要軟體架構？
Uncle Bob 在書中提到了這個概念，使用軟體架構是為了最小化建置和維護系統時所需要的成本。

“The goal of software architecture is to minimize the human resources required to build and maintain the required system.” ― Robert C. Martin, Clean Architecture

在執行專案時，我們都會希望開發的時程越短越好，但如果在開發時沒有去運用架構，僅僅只是完成需求上的功能就好，就有可能將這些忽略的開發成本無形地轉嫁到後續的維運成本。
舉例來說，一個已經上線的專案突然要改變 UI，如果沒有設計好架構，可能就要連動到商業邏輯的修改，導致你在維運階段就需要花更多時間去測試和滿足需求。
因此我們會需要了解簡潔架構，簡潔架構會透過三個原則，分別是相依性原則、分層原則和跨層原則來達到關注點分離，也就是說系統核心不會知道外部所使用資料庫、框架和 UI 是甚麼，只需要關注本身的商業邏輯。後續即便有需求變更，也可以更快速聚焦和修改，維運成本也隨之降低。

要特別提到的是這邊提到的三大原則並沒有在 Clean Architecture 書中所定義，而是先前我在上Teddy的課程中，他對於Clean Architecture全書做的總結，因我覺得這樣能更容易理解簡潔架構，所以在這篇文章中會使用三大原則來說明簡潔架構

來源請參考: https://teddy-chen-tw.blogspot.com/2020/08/clean-architecture.html

使用簡潔架構的優勢

  • 獨立框架

    將框架當作工具來使用，而工具是可以被替換的，也就是系統內層不會依賴於框架

  • 獨立 UI

    UI的變動不會影響到商業邏輯，因 UI 時常會因滿足客戶需求而進行變動

  • 獨立 Database:

    將資料庫也作為工具使用，舉例來說，原先使用 MySQL，但後續想要改使用 MongoDB 也能直接替換卻不會影響到商業邏輯。套用到其他中間件也是，像是 Message Queue，一樣也能將 kafka 或 RabbitMQ 當作工具來使用

  • 獨立外部服務:

    將商業邏輯與外部系統切開

  • 可測試:

    商業邏輯可以在沒有 UI、資料庫和其他中間件等等的情況下進行測試

簡潔架構三原則

分層原則

那在簡潔架構中，主要我們會分成4層，而每一層都有其負責的用途及功能：

  • Entitiy

    商業邏輯層，Entity 可以是一個物件，也可以是一組資料結構，Entity 不會受到外部的影響(像是DB或UI)，關注的重點只有他自己。

  • UseCase

    使用案例層，系統中所有的使用案例都會封裝在這一層，控制 entities 資料的流動，並在這層透過依賴反轉去操控外部系統 (e.g. DB) 去滿足使用案例。舉例來說，建立、修改及刪除會員都可以個別視為一個 UseCase。

  • Interface Adapter

    接口轉換層，負責資料的接收和資料格式的轉換。舉例來說，當會員資料透過 RESTful API 傳送到系統，就會由這一層接收，並且做轉換再傳入到 UseCase 層。

  • Framework & Driver

    負責框架、DB 和外部服務的相關設定及實作

相依性原則

從上面的圖可以了解到簡潔架構是由四個同心圓組成，而越往內層，層級就越高。最內層是處理商業邏輯的核心，而外層就偏向於處理流程上的機制，因此程式碼的依賴只能由外往內。簡單來說，內層的程式碼不能引入外層的變數、函式或類別等等，這就是相依性原則。

跨層原則

簡潔架構在做跨層資料的輸入或輸出時，都會使用基本的資料結構來傳遞，也就是將原先的物件再重新包裝程一個純資料結構的物件再傳送，這樣可以避免違反相依性原則。

舉例來說，當建立會員時，會先由 Adapter 層接收 Restful API 的請求，而當我們需要調用 UseCase 層去新增會員時，就會需要先建立一個純POJO在UseCase層讓Adapter進行轉換，藉此UseCase層就不會引用到外層的東西。

結論

在以往的開發過程中，如果我們使用MVC架構，因為沒有相依性原則和跨層原則的規範，就會是直接建立實作類別。而在缺少了介面定義的情況下，單元測試就會相對難寫。另外在沒有單元測試的情況下，維護上就容易出現改Ａ壞Ｂ，增加維運的成本。所以我們可以透過簡潔架構來將系統進行分層，將商業邏輯和外部系統、框架做一個解耦，一方面是我們能更彈性的去抽換中間件，另一方面則是在測試上更容易可以去撰寫。

Angular Service

發表於 2022-01-05 更新於 2024-01-06 分類於 Angular

前言

廣義的來說，服務可以是任何值、函式或是特性，
但通常來說，服務會定義成類別 ( class )，用來處理特定的事情和邏輯。
Angular將元件和服務區分開，
元件定義為處理畫面操作的角色，像是屬性綁定、事件綁定等等，
而像是向後端取得資料、檢核輸入或撰寫Log等等動作，
就會定義在服務裡面，再將服務注入到元件裡面使用。
如此一來，多個元件可以依賴同一個服務，
就能達到重複利用和模組化的效果，
另外也能讓元件輕量化，程式碼更整潔。

目錄

  • 依賴注入
  • 建立服務
  • 提供服務

依賴注入

Dependency Injection 是指依賴注入，簡稱就是 DI。
元件會根據需求，將所需要的服務注入到元件裡面，
而元件就能得到操控服務的權限。
以電商平台的例子來說，
產品元件和購物車元件，都會需要取得產品資訊，
這時候就能將取得產品資訊的邏輯定義在服務裡面。

建立服務

在Angular中，如果要將一個類別定義為服務，需要用到 @Injectable 裝飾符，
以便讓Angular將它視為依賴並注入到元件裡面。

  1. 注入器是主要的機制，Angular會在啟動專案時自動幫你建立，所以不需要自己建立注入器。
  2. 注入器會建立管理依賴的容器，讓它們可以被重複利用。
  3. 提供者會是一個物件，用來告訴注入器如何獲取或建立依賴。

以官方的Hero Service為例，
有一個Hero元件，是用來顯示英雄清單的資料，
而資料則是透過Hero Service所取得。
因此在元件被實體化時，須將此依賴注入到元件裡面。

這時候注入器就會在容器尋找是否已存在Hero Service這一個依賴，
如果存在，則將此依賴開放給元件使用；
反之，就會在容器裡面新增依賴並開放給元件使用。

提供服務

你必須為服務註冊一個提供者(Provider)，這樣服務才能被使用。

提供者可以被定義在服務的裝飾符裡面，
而 root 則表示此服務在整個專案裡面只會擁有一個，
每個元件透過注入來分享這一個服務。

備註: 倘若使用 AngularCLI 建立服務，就會自動設定為 root。

當然也可以將服務設定為特定模組所使用，

表示這個服務會被模組內的元件所共享。
又或者是特定元件，

表示這個服務在該元件內所共享。
總結上述，服務的提供者可以為

  • root - 根等級，表示整個專案

Angular Pipe

發表於 2022-01-05 更新於 2024-01-06 分類於 Angular

前言

我們可以使用 Pipe 來轉換字串、貨幣、日期或其他資料的顯示，
簡單來說，你可以將 pipe 視為一個function在Template裡面使用，
input → pipe → output
舉例來說，
當日期的資料為 西元 2021/05/10，
就可以透過 pipe 轉換為 民國 110/05/10。

常用的 Pipe

  • Date

    將日期轉換為本地格式

  • Upper Case

    英文字母轉換為大寫

  • Lower Case

    英文字母轉換為小寫

  • Currency

    將數字轉換為貨幣字串，格式也能在地化

  • Decimal

    將數字轉換為有小數點的字串，格式也能在地化

  • Percent

    將數字轉換為有百分比的字串，格式也能在地化
