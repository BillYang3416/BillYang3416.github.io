在 Golang 操作 PostgresSQL

發表於 2024-01-17 更新於 2024-01-21 分類於 Golang ， Postgres

前言

本篇文章主要會說明在 Golang 中操作 PostgresSQL 的套件及其方法，
首先會先說明原生SQL和 ORM 的差別以及介紹相關的開源套件，
接著就會針對 pgx 這個套件作更進一步的說明和介紹，
那因爲 pgx 是使用原生SQL去操作 PostgresSQL，
所以最後會說明如何搭配 squirrel 使用去組出 pgx 所需要的原生SQL。

  • 套件的選擇

套件的選擇

當我們要在 Golang 中連接 PostgresSQL 時，
有幾種套件可以挑選，那我們先將這些套件分成兩類在一一說明，
第一個是 Drivers，當我們要對資料庫做新刪查改時都是使用基本的 SQL 語句，也就是 Plain SQL。
另一個就是 ORM (Object-Relational-Mapping)，
在物件導向的語言中，通常都會有 ORM 的套件來將物件轉換成資料庫的資料，
這中間不需要開發者組出SQL語句，而是透過套件來組成，像是 JAVA 中的 Hibernate。
而 Golang 雖然並非純物件導向的語言，但一樣也有第三方套件提供了這種作法。

Drivers (Plain SQL)

pq

lib/pq

pq 是連接 PostgresSQL 的 driver 之一，
通常會與標準函式庫的 database/sql 搭配使用。
優點就是簡單上手且可隨時切換成不同的資料庫，不會被 PostgresSQL 所綁住，
但缺點就是效能頗差。
另外值得一提的是該套件的開發者已經宣告停止出新的功能，
目前處於 maintain 的狀態，而開發者則推薦使用 pgx 作為連接 PostgresSQL 的 driver

程式碼取自 go pkg

pgx

jackc/pgx

如果要在 Golang 中操作 PostgresSQL，可以說 pgx 應該是目前最多人推薦的選擇，
他有幾個優勢，首先要說的就是效能，他的效能比 pq 或者是 Gorm 都還要好，
並且他也有支援 database/sql 的用法，
也就是說我們也可以像前面 pq 的程式碼一樣去使用 database/sql 去下語法，
但實際與 PostgresSQL 對接的還是 pgx。
再來，最重要的是 pgx 他還支援一些 PostgresSQL 特性的實現，
而這一些都是 pq 和 database/sql 沒辦法滿足的需求。
但缺點就是當要切換不同的資料庫時，會比較難改動，
再來就是 pgx 相對 pq 複雜，需要多一點時間學習。
在後續的文章中，我會在介紹關於 pgx 的一些使用方法。

ORM (Object-Relational-Mapping)

GORM

雖然 Golang 並非純物件導向的語言，但因爲有 struct、method 和 interface，
所以我們可以使用 struct 來對應 database 中的 table。
而 GORM 就是提供了這樣的方法來實現 ORM，並且他還能自動建立新的 schema 和 table，
也能改變舊有 table 的結構。
此外對於資料庫的新刪查改也是能非常簡單地去實現。
因此可以說他的優點就是開發上相對簡單，並且不受限於不同資料庫的限制，可以隨時無痛轉換，
而他的缺點大概有兩個，其一是效能，因為我們是透過套件去產出SQL語句，
相對效能一定會比直接下原生SQL來得差，
其二就是無法寫出複雜的 Queries，碰到這類情況時，可能還是需要使用原生的SQL來解決。

選擇的建議

那在介紹完上述的套件之後，
我認為沒有哪個套件一定是最好，
假設今天你的系統只會有簡單的CRUD並且也不追求效能的話，
那其實就可以選擇使用 GORM 來減少寫原生SQL的時間，同時也能降低開發者的學習成本。
但如果另一個系統非常追求效能並且有很多複雜的 Queries 時，
就應該改使用 pgx 寫原生SQL，相對的時間成本也會提高，
所以依據系統的需求和情境來選擇套件才會是最佳解。

連接 PostgresSQL

使用 pgx 連接 PostgresSQL 的方式可以分為兩種，
第一種就是使用 pgx.Connect，這種方式只會與資料庫建立一條連線，

淺談關聯資料庫的 ACID

發表於 2024-01-08 更新於 2024-01-12 分類於 Database

前言

在本篇文章中，我們會先介紹甚麼是 transaction，
因為 transaction 是 ACID 的關鍵概念，必須先理解 transaction 才能懂這4個特性要達到的目的，
而這4個特性分別是 Atomicity、Isolation、Consistency 和 Durability，
也是關聯式資料庫運作時所遵循的原理。

目錄

  • Transaction
  • Atomicity
  • Isolation
  • Consistency
  • Durability

Transaction

甚麼是 Transaction ?
Transaction 在關聯式資料庫的概念就是一組 DML (Data Manipulation Language) 的集合，
像是我們平常看到 Insert、Update、Delete 和 Select，
將這些排列組合起來後就可以稱為一個 Transaction。
以銀行轉帳為例，假設大明想要轉帳一百塊給小王，
那銀行就會先查詢出大明的帳戶，接著就會先扣除大明帳戶的一百塊，最後將小王的帳戶增加一百塊。
在這樣的流程裡，其實就包含3個 SQL Queries，分別是 Select、Update 和 Update，
將其組合起來就會是一個 Transaction。

另外 Transaction 從開始到結束，大概有這幾種狀態:

  • Begin (開始)
  • Check Constraints (確認條件)
  • Execution of SQL Queries (執行)
  • Commit / Rollback (生效或回滾)

接下來我們在用圖來示範轉帳的案例會讓大家更清楚些，
在正常的情況下，因為大明的帳戶有超過 200 塊，所以最後 Transaction 就會 Commit，
代表這次所執行的 SQL Queries 都成功，轉帳也會順利轉進各自的帳戶。

但假設說今天大明的帳戶沒有超過 200 塊，只有 50 塊，
當 Transaction 要執行 Update Account1 的時候就會發生錯誤，
因為沒有滿足 Check Constraints (BALANCE > 200)，接下來就會直接執行 Rollback 代表轉帳失敗。

在了解 Transaction 之後，接著我們來看四大特性，分別是
Atomicity、Isolation、Consistency 和 Durablilty。

Atomicity

Atomicity refers to all or nothing.

所謂的 Atomicity 就是在一個 transaction 中，所有執行的 SQL Queries 要全部成功或全部失敗，
也就是說只要有一個 rollback，即便它已經是最後一個執行的 SQL Query，
前面所成功的 SQL Queries 也都要 rollback。
這邊我們一樣拿轉帳的例子來說明，假設大明的帳戶已經被扣款了，
而就在這中間幾秒的時間，DB被強制關閉了，小王的帳戶餘額並沒有增加，
如果不遵循 Atomicity，這一百塊就會不見，對於銀行客戶來說就會是一個很嚴重的問題，
但如果遵循 Atomicity，DB 做了 rollback，這樣資料就會存在，對我們的影響就是重做一次交易而已。
因此 Atomicity 相當重要，一來是能確保資料的一致性，二來是在發生錯誤時，也能夠保護資料不受到影響。

Isolation

接下來我們來了解 Isolation，
大家先試想看看，假設我們架設了一個電商網站，
是不是就會有許多的 User 在同一時間會對商品瀏覽、下單，
而這時候網站的前後端就會開始資料交換，而後端就要向 DB 取得資料或異動資料。
這就代表 DB 的 Transaction 大多都會是 Concurrent（並行），
在這樣的狀況下，一個 transaction 被另一個 transaction 影響後所產生的現象，
我們就會稱作 Read Phenomena。
而要解決 Read Phenomena 的問題，
就要使用 Isolation Level 來確保資料的正確性和一致性，
讓 transaction 不會受到 Concurrent 的影響，獨立的執行 SQL Queries。

Read Phenomena

在 Read Phenomena 中，有四種現象分別是 Dirty Read、Non-repeatable Read、Phantom Read 和 Lost Update。

Dirty Read

當我們在一個 Transaction 中，查詢到其它 Transaction 尚未 Commit 的資料異動，
在還沒 Commit 之前，這種資料異動都屬於髒資料，代表還未生效，這種現象就會稱為 Dirty Read。
這邊我們用兩個 transaction 來解釋這個現象，
假設我們有一個產品銷售的資料表，首先 TX1 會執行兩次 Select 指令，
在執行第二次之前，TX2 會先改變產品1的銷售數量，藉而影響 TX1 的第二次查詢結果，
如果 TX2 並沒有 Commit 而是 Rollback，那 TX1 所產生的資料就是錯誤的。

Non-repeatable Read

在一個 transaction 中對同一筆資料做了兩次查詢，而查詢結果卻不相同，
這是因為受到另一個 transaction 對相同的資料做 Commit 讓同筆資料異動生效，
而這就是 Non-repeatable Read 的現象。
這邊我們一樣用兩個 transaction 來解釋這個現象，
假設我們有一個產品銷售的資料表，首先 TX1 會執行兩次 Select 指令，
在執行第二次之前，TX2 會先改變產品1的銷售數量且 Commit，藉而影響 TX1 的第二次查詢結果。

Phantom Read

在一個 transaction 中對相同資料做了兩次範圍的查詢，而查詢結果卻不相同，
這是因為受到另一個 transaction 對相同資料做了新增或刪除並且 Commit，
而這就是 Phantom Read 的現象。
這邊我們一樣用兩個 transaction 來解釋這個現象，
假設我們有一個產品銷售的資料表，首先 TX1 會執行兩次 Select 指令，
而這兩個指令都是 ragne query。
在執行第二次之前，TX2 會先新增產品3且 Commit，藉而影響 TX1 的第二次查詢結果。
Phantom Read 和 Non-repeatable Read 的差別就在於：
Phantom Read 通常是發生在資料表有被新增或刪除，
而 Non-repeatable Read 則是同筆資料有被異動的情況下發生。

Lost Update

在一個 transaction 中，我們先對一筆資料做異動，
然而另一個 transaction 也同時對同筆資料做異動且 Commit，
這個就會導致第一個 transaction 在查詢同筆資料時，
得到的結果不是自身 transaction 所異動的資料，而這就是 Lost Update。
這邊我們一樣用兩個 transaction 來解釋這個現象，
假設我們有一個產品銷售的資料表，首先 TX1 會先對產品2執行一次 Update 指令，
接下來 TX2 同時對產品2也執行 Update 指令且 Commit，
藉而就影響 TX1 的第二次查詢，銷售總額不是 TX1 調整數量的結果。

Isolation Level

前面我們說明了四種在 transaction 並行會發生的現象，
也了解到每一種現象發生時可能都會導致資料不一致或資料遺失的狀況，
這個絕對是我們需要去避免的，
因此資料庫會使用 Isolation Level 來解決 Read Phenomena。

下圖取自維基百科，這張表格完美了詮釋各個 Isolation level 解決了哪些現象，

  • Read Uncommitted

    在 transaction 執行期間，可以拿到其他 transaction 尚未 commit 的資料，
    而這就無法避免發生 Dirty Read

  • Read Committed

    在 transaction 執行期間，可以拿到其他 transaction commit 後的結果，
    但前面提到的 Non-repeatable Read 或 Lost Update 就有可能發生。
    另外大多的資料庫都會使用這個作為預設的 Isolation Level

  • Repeatable Read

    當 transaction 開始執行時，如有資料被讀取到，資料庫就會將其鎖住，避免其他 transaction 對同筆資料做異動，而這就不會發生 Non-repeatable Read 和 Lost Update

  • Serializable

    前面提到的 Repeatable Read 可能只是鎖住單個 Row，
    但如果遇到資料表被新增或刪除時，還是無法避免 Phantom Read。
    因此 Serializable 可以針對單個 Row 、範圍性的 Query 或是整張表給上鎖，
    當 transaction 執行期間，其他 transaction 必須等到鎖釋放後才能異動資料。
    在這種情況下，資料雖然最安全，但效能也相對差勁，因此不太常被使用

最後有一個重點要注意，就是不同的資料庫在 Isolation Level 所使用的機制不盡相同，
像 Postgres 在 Repeatable Read 就使用了 Snapshot 的機制，
Snapshot 是指當 transaction 開始執行前，會將要查詢的資料做一個快照，
而快照在執行期間不會有變動，因此就不會發生 Phantom Read 的現象。
所以在使用 Isolation level 時，記得要考慮到資料庫的特性。

Consistency

接下來我們要來介紹 Consistency，
所謂的 Consistency 就是在 transaction 執行時，資料庫要確保資料是符合 Constraint (限制條件)，
藉此讓資料維持一致性和正確性。
常見的 Constraint 大概有這幾種：

  • Data Type

    資料必須符合 Schema 所規範的資料型別

  • Range

    在前面帳戶轉帳的例子，我們確認帳戶在轉帳前是否有超過兩百塊，
    這一種就算是 Range Constraint

  • Uniqueness

    唯一性，代表該鍵值在整張資料表中只能被一筆資料所擁有

  • Referential Integrity

    當兩張 table 要建立關聯時，我們會使用 foreign key 去建立兩張表的關係。
    這邊可以舉一個例子，在餐廳裡，我們可能會有客戶和訂單的資料表，那客戶中就會存放客戶ID、聯絡資訊和姓名等等，
    至於訂單一定是有客戶才會產生的，因此我們會在訂單的資料表設定客戶ID作為訂單的 FK。
    接下來假設我們要新增訂單的資料時，他就會確認訂單的客戶ID是否有出現在客戶的資料表，
    如果沒有就代表客戶不存在，這筆訂單就會建立失敗，
    反之，如果我們要刪除客戶的資料時，也必須確認訂單中都沒有這個客戶ID。

Durability

前面提到的 Atomicity、Isolation 和 Consistency 基本上都是確保 trsanction 執行期間的正確性，
那最後要提到 Durability 則是要確保當 transaction commit 後，
資料的狀態就會是 persistent，並且必須存進 non volatile 的儲存位置，像是硬碟，
即使遇到關機或錯誤等狀況，資料也都要能還原再儲存。
而通常資料庫會使用 WAL(Write-Ahead Logging)的機制來確保 Durablilty，
也就是說每筆資料的異動一定都會先寫下 Log，
因此即便在 transaction commit 後，資料的異動尚未 flushing 到儲存位置(硬碟)時，
這時候如果 DB 突然被關閉了，還是能透過 Log 來進行還原，藉此確保了 Durablilty。

結論

在本文中，我們先說明了何謂是 transaction，因為可以看到 ACID 每一個特性都和 transaction 有關，
所以必須先理解 transaction 再來談四個特性。
Atomicity 確保 transaction 的 SQL Queries 必須全部成功或全部失敗，
而 Isolation 則是確保 transaction 的獨立性，避免受到並行 transaction 的影響導致資料不一致或遺失，
再來 Consistency 則透過條件限制 transaction 的執行，確保資料的一致性和正確性，
最後 Durablity 則是確保 transaction commit 後，資料必須存入不受電力影響的儲存位置像是硬碟，
並且在遇到關機或錯誤等狀況時，也會有 WAL 的機制讓資料可以還原並儲存。
總結來說，在資料庫中 ACID 是非常重要的概念，也是關聯式資料庫運作時所遵循的原理。

如何設計微服務的共用函式庫

前言

本篇文章首先從思考問題出發，探討為甚麼我們需要在微服務中使用共用函式庫，
以及使用共用函式庫的優缺點，最後則說明設計時可以遵循的 Best Practices。

目錄

  • 為甚麼要設計共用函式庫
  • 使用共用函式庫的優點和缺點
  • 設計共用函式庫的 Best Practices

為甚麼要設計共用函式庫

在微服務的架構當中，我們會將一個系統切分成好幾顆服務，
以電商系統為例，我們可能就會切分成訂單、庫存和交易等服務，
而在撰寫這些服務時會出現通用的功能，像是日誌。
如果每一顆服務都要額外去寫日誌功能，就會違反 DRY(Don't Repeat Yourself) 的精神，
也會讓各團隊出現重工的情況，浪費了大量的時間成本。
透過設計這些微服務會使用到的共用函式庫，讓多顆服務可以複用，
一來確保程式碼不會重複，二來也能保持服務使用第三方套件的一致性。
藉此來加快團隊的開發效率。

使用共用函式庫的優點和缺點

優點

  • 不會出現重複的程式碼
  • 減少團隊間的重工，避免浪費人力和時間成本
  • 各服務能保持對第三方套件的一致性，避免發生版本間的衝突
  • 當有 Bug 出現時，只需要修改共用函式庫；不需要手動改每一顆服務

缺點

  • 過於依賴共用函式庫，會導致各服務之間的耦合度增加
  • 當共用函式庫更新時，所有服務可能都會需要同步調整版本和程式碼

設計共用函式庫的 Best Practices

在設計共用函式庫時，有一些 Best Practices 可以遵循:

  • 弄清楚共用函式庫的需求和功能，讓其符合單一責任的原則避免多餘的程式碼
  • 設計時要專注在功能上，但與服務的耦合度要低，這樣共用函式庫改動時就能減少對服務的影響
  • 在共用函式庫中多使用介面去增加配置的靈活性，一是方便函式庫的擴展，二是讓服務也能客製函式庫的功能
  • 共用函式庫要能讓團隊簡易使用和上手，減少學習的時間成本
  • 更新函式庫應考慮兼容性，避免服務因函式庫的影響而改動程式碼

結論

在微服務中設計共用函式庫，其主要目的就是為了讓各服務能更專注於自身的商業邏輯，
一是減少程式碼重複的現象，二是確保服務使用第三方套件一致性。
但在設計共用函式庫就必須考慮和服務的耦合度，將改動的影響最小化，
避免服務因為函式庫的更新而調整，反而增加團隊的時間成本和溝通成本。

最後則簡單介紹非同步的通訊協定，像是 AMQP。

  • 甚麼是同步和非同步
  • 在微服務中使用非同步的優勢

甚麼是同步和非同步

以往有接觸過 Web Service 的朋友，應該都對同步和非同步的概念並不陌生，
這邊我們舉一個例子來說明同步和非同步的差別，
假設在電商系統中，我們有兩個服務分別是訂單和庫存，
訂單服務接收到前端建立訂單的 HTTP 請求，當訂單成立時，我們需要去減少庫存的商品數量，
這時候訂單服務會再發送 HTTP 請求給庫存服務，庫存對資料庫操作後再回應給訂單服務，
訂單服務再回應給前端表示建立訂單成功，這個資料流程就會為同步。
透過下方的圖示可以看到訂單和庫存服務已經耦合在一起，並且前端接收到回應時間也會被拉長。

所以我們可以使用非同步的機制來將服務間的關係解耦並且減少回應時間，
透過 Message Broker 建立生產者和消費者的 Pattern，
生產者可以持續將訊息放入到 Message Broker，再由消費者透過訂閱的方式向 Message Broker 取得訊息在進行處理，這種資料流程就會為非同步。

在微服務中使用非同步的優勢

那為甚麼我們需要在微服務中使用非同步的機制呢?
其實從前面的例子可以了解到幾個優勢:

  • 解耦服務之間的依賴關係

    假設我們使用同步的方式處理資料，就會遇到當庫存服務改變 API 的 URL 時，連帶就會影響到訂單服務；使用非同步能減少服務間的直接依賴

  • 增加效能

    因服務間的資料流不會被同步所阻塞，所以回應時間就會與之降低，增加系統的效能

  • 容錯率變高

    因服務之間不會直接依賴，所以當庫存服務壞掉時並不會影響到訂單服務的流程，所造成的影響可能只是 Message Broker 的 Queue 會儲存大量的訊息

因為這些優勢，所以在微服務的架構中都會設計非同步的資料流和 Message Broker 的角色。

Advanced Message Queuing Protocol (AMQP)

在微服務的架構中，通常我們會使用 AMQP 作為非同步的通訊協定，
AMQP 也和 HTTP 一樣，屬於 OSI 中的 Applicaion Layer，
通常會被 Message-Oriented 的 Middleware 所使用，用來和服務做資料交換。
而使用 AMQP 的 Middleware 主要有兩個，分別是 RabbitMQ 和 ActiveMQ。

當然在微服務中也會使用其他的 Middleware 像是 Apache Kafka、Amazon SQS 和 Google Pub/Sub，這些都沒有支援 AMQP，但一樣能達到非同步的目的和取得它的優勢。

結論

透過這篇文章的說明，可以了解到非同步機制的設計在微服務中不可或缺，
對此有興趣的讀者可以在深入了解 AMQP 的原理和 middleware 的使用方法，
像是 RabbitMQ 的官方文件就相當完整，在參閱後就能簡單建立出具有非同步機制的服務。

Concurrency

Concurrency is not parallelism.

接下來我們就來透過例子來了解什麼是 Parallelism 和 Concurrency，
這個例子是量販商的工作人員正在透過推車將電商貨品搬到貨車上。

Parallelism

假設到了過年期間，貨量突然暴增，店長覺得這樣出貨的速度太慢，

Concurrency

那我們換一個想法，當我們不增加一個出貨站和一台貨車時，我們能怎麼做呢？

這就是 Concurrency。

透過上述的例子可以看到 Parallelism 是通過加大資源在同一時間平行化地來執行任務，

深入解析 RabbitMQ 的參數設定

發表於 2023-12-22 更新於 2024-01-06 分類於 RabbitMQ

前言

當我們在建立 RabbitMQ 的物件時，像是 Exchange、Queue 或 Binding，
我們可能會需要設一些參數上去，像是 message-ttl 或 max-length 等等。
而這些參數通常會需要在 runtime 期間可以被改動的，
所以我們會使用 Policy 來配置這些參數，這也是 RabbitMQ 官方所推薦的作法。
這篇文章會介紹這些參數的用途和如何使用 Policy 進行配置。

以 Queue 來說，如果建立時有綁定 message-ttl 就代表無法更改值，除非將 Queue 砍掉在重建。

目錄

  • Exchange
  • Queue
  • Binding
  • 閥值類參數的建議

Exchange

Exchange 的參數可以劃分為靜態參數和可選性參數，
而可選性參數都建議都使用 Policy 來配置，
因為如果建立時就配置可選性參數，就會需要刪除物件才能修改參數的設定。

靜態參數

名稱 用途 預設值
type Exchange 的類型，像是 direct、fanout、topic 和 header
durable broker 重新啟動後仍然存在
auto_delete 當沒有 Queue 與 Exchange 做 binding 時會自動刪除 false
internal Exchange 不能接收發送者的訊息，只能在 broker 內部用來交換訊息 false

可選性參數

名稱 用途 預設值
alternate-exchange 當訊息無法被轉到對應的Queue時，訊息可以轉到替代的 Exchange

Queue

至於 Queue 一樣也可以分為靜態參數和可選性參數，
而可選性參數都建議都使用 Policy 來配置，
因為如果建立時就配置可選性參數，就會需要刪除物件才能修改參數的設定。

靜態參數

名稱 用途 預設值
durable broker 重新啟動後仍然存在
auto_delete 當沒有 Binding 也沒有消費者連接時就會自動刪除 false

可選性參數

名稱 用途 預設值
x-message-ttl 訊息能在 Queue 裏存活的時間，如果時間超過，訊息就會被轉至到 DLQ 或刪除。另外如果訊息的狀態已經變成　unacknowledge 就不會計算 ttl
x-expires 如果 Queue 在時間內都沒有被使用到，則 Queue 會被自動刪除
x-max-length 訊息在 Queue 裏的最大存放數量，如果訊息的狀態已經變成　unacknowledge 則不在此列
x-max-length-bytes 訊息在 Queue 裏的最大存放空間，如果訊息的狀態已經變成　unacknowledge 則不在此列
x-overflow 當 x-max-length 或 x-max-length-bytes 的條件達成時，Queue 對接收的訊息所做出的行為，可以分為 drop-head (最舊的會被移除)、reject-publish (發送端會發送失敗) 和 reject-publish-dlx (發送失敗被轉移到 DLQ) drop-head
x-dead-letter-exchange 當訊息被 rejected 或過期時，要經由哪一個 Exchange 做轉移
x-dead-letter-routing-key 當訊息被 rejected 或過期時，用來轉移 DLQ 的鍵值
x-queue-mode 訊息要儲存在 Memory 還是 Disk，可以分為 default(memory) 和 lazy(disk) default
x-queue-type Queue 的類型，可以分為 classic 和 quorum classic

Policy 額外能設定的參數

名稱 用途 預設值
delivery-limit 當訊息沒有被 RabbitMQ 傳遞成功時，RabbiMQ 嘗試傳遞的最大次數，避免發生 Poison Message 的現象，也就是當消費者消費訊息失敗，訊息回到 Queue，RabbitMQ又在傳遞給消費者，如此重複下去
dead-letter-strategy 當訊息傳遞失敗或超過TTL，在傳遞給DLQ的策略，可以分為 at-most-once(訊息最多傳遞一次給DLQ) 和 at-least-once(訊息至少要傳遞給DLQ成功一次，必須搭配 overflow 的 reject-publish) at-most-once

Binding

Binding 是用來配置 Exchange 和 Queue 的關係，
這一塊通常不需要使用 Policy 來配置可選性參數，
只需要在建立的時候指定 Exchange、Queue 和 RoutingKey。

靜態參數

名稱 用途 預設值
Exchange Exchange 的名稱
Queue Queue 的名稱
Routing Key 路由的鍵值

設置閥值類參數的建議

前面我們提到了 Queue 的可選性參數，其中有三個參數分別是 x-message-ttl、x-max-length 和 x-max-length-bytes，
這三個可以歸類成閥值類的參數，代表我們需要依據系統來動態調整這些參數。
這邊提供一些調整參數時的準則當作參考:

  • x-message-ttl
    • 消費者行為: 如果消費者消費比較慢，則要將 ttl 設長一點
  • x-max-length
    • 吞吐量大小: 計算 publising rate 和 consuming rate，如果生產者產製訊息比消費者更快，就需要更大的 x-max-length
  • x-max-length-bytes
    • 訊息大小: 如果每一筆訊息的容量都偏大，就可以將 x-max-length-bytes 調高，以便負荷更多的訊息

結論

在這篇文章我們詳細介紹了關於 Exchange、Queue 和 Binding 的參數，
並且還區分了靜態參數和可選性參數。
另外值得一提的是，使用 Policy 來配置可選性參數相當重要，
因為在系統運行時，我們不希望要刪除物件來修改參數，
一旦刪除了物件，不只會影響到服務的運行，還要重新配置 Exchange 和 Queue 的 Binding，一不小心就很容易出錯。
最後則提供了一些配置閥值類參數的建議。
