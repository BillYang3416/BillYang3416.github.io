在 Golang 操作 PostgresSQL

發表於 2024-01-17 更新於 2024-02-10 分類於 Golang ， Postgres

前言

本篇文章主要會說明在 Golang 中操作 PostgresSQL 的套件及其方法，
首先會先說明原生SQL和 ORM 的差別以及介紹相關的開源套件，
接著就會針對 pgx 這個套件作更進一步的說明和介紹，
那因爲 pgx 是使用原生SQL去操作 PostgresSQL，
所以最後會說明如何搭配 squirrel 使用去組出 pgx 所需要的原生SQL。

目錄

  • 套件的選擇
  • 如何使用 pgx
  • 使用 squirrel 組出 SQL Query

套件的選擇

當我們要在 Golang 中連接 PostgresSQL 時，
有幾種套件可以挑選，那我們先將這些套件分成兩類在一一說明，
第一個是 Drivers，當我們要對資料庫做新刪查改時都是使用基本的 SQL 語句，也就是 Plain SQL。
另一個就是 ORM (Object-Relational-Mapping)，
在物件導向的語言中，通常都會有 ORM 的套件來將物件轉換成資料庫的資料，
這中間不需要開發者組出SQL語句，而是透過套件來組成，像是 JAVA 中的 Hibernate。
而 Golang 雖然並非純物件導向的語言，但一樣也有第三方套件提供了這種作法。

Drivers (Plain SQL)

pq

lib/pq

pq 是連接 PostgresSQL 的 driver 之一，
通常會與標準函式庫的 database/sql 搭配使用。
優點就是簡單上手且可隨時切換成不同的資料庫，不會被 PostgresSQL 所綁住，
但缺點就是效能頗差。
另外值得一提的是該套件的開發者已經宣告停止出新的功能，
目前處於 maintain 的狀態，而開發者則推薦使用 pgx 作為連接 PostgresSQL 的 driver

程式碼取自 go pkg

pgx

jackc/pgx

如果要在 Golang 中操作 PostgresSQL，可以說 pgx 應該是目前最多人推薦的選擇，
他有幾個優勢，首先要說的就是效能，他的效能比 pq 或者是 Gorm 都還要好，
並且他也有支援 database/sql 的用法，
也就是說我們也可以像前面 pq 的程式碼一樣去使用 database/sql 去下語法，
但實際與 PostgresSQL 對接的還是 pgx。
再來，最重要的是 pgx 他還支援一些 PostgresSQL 特性的實現，
而這一些都是 pq 和 database/sql 沒辦法滿足的需求。
但缺點就是當要切換不同的資料庫時，會比較難改動，
再來就是 pgx 相對 pq 複雜，需要多一點時間學習。
在後續的文章中，我會在介紹關於 pgx 的一些使用方法。

ORM (Object-Relational-Mapping)

GORM

雖然 Golang 並非純物件導向的語言，但因爲有 struct、method 和 interface，
所以我們可以使用 struct 來對應 database 中的 table。
而 GORM 就是提供了這樣的方法來實現 ORM，並且他還能自動建立新的 schema 和 table，
也能改變舊有 table 的結構。
此外對於資料庫的新刪查改也是能非常簡單地去實現。
因此可以說他的優點就是開發上相對簡單，並且不受限於不同資料庫的限制，可以隨時無痛轉換，
而他的缺點大概有兩個，其一是效能，因為我們是透過套件去產出SQL語句，
相對效能一定會比直接下原生SQL來得差，
其二就是無法寫出複雜的 Queries，碰到這類情況時，可能還是需要使用原生的SQL來解決。

程式碼取自 GORM Guides

選擇的建議

那在介紹完上述的套件之後，
我認為沒有哪個套件一定是最好，
假設今天你的系統只會有簡單的CRUD並且也不追求效能的話，
那其實就可以選擇使用 GORM 來減少寫原生SQL的時間，同時也能降低開發者的學習成本。
但如果另一個系統非常追求效能並且有很多複雜的 Queries 時，
就應該改使用 pgx 寫原生SQL，相對的時間成本也會提高，
所以依據系統的需求和情境來選擇套件才會是最佳解。

如何使用 pgx

接下來我們來了解如何使用 pgx：

建立專案

在使用之前，我們先使用兩個指令來建立專案和載入套件。

連接 PostgresSQL

使用 pgx 連接 PostgresSQL 的方式可以分為兩種，
第一種就是使用 pgx.Connect，這種方式只會與資料庫建立一條連線，
適合用於簡單的專案，像是 script 或範例。
但如果專案是 web service 的性質，那就非常的不適合，
因為同一時間可能會有多個使用者發請求到後端，
而後端同時有多個 Concurrent 的請求要對資料庫做異動時，就會被卡住。
因此通常 web service 都會推薦使用第二種方式，也就是 pgxpool，
這個方式會建立多條和資料庫的連線，藉此來滿足多個請求對資料庫的操作。

想多了解 Connection Pooling 可以參考 What is database pooling?

pgx.Connect

下面程式碼是我們在使用 pgx.Connect 的範例，
有幾個地方可以來說明一下，

  • contexnt.Background()

    context 是標準函式庫之一，主要是用來對執行的進程做取消或是加上timeout，這邊可以舉一個 request 的例子，

  • defer conn.Close(…)

    每當我們向資料庫建立連線時都需要記得關閉，這和開檔和關檔的道理是一樣的，避免資源的耗用。
    至於 defer 則代表當 main function return 後才會執行，因為程式對資料庫的操作已經結束了，不需要再保持連線。
    如果沒有加上 defer 就會導致連線建立後就馬上關閉

第一種就是使用 pgxpool.New，用法和 pgx.Connect 類似。

而另一種就是使用 pgxpool.NewWithConfig，
這個用法的優勢就在於彈性，
當你今天想要在程式決定最大或最小的連線數又或者是在連線的建立前後去做一些事情都可以透過 config 去設定。

Tx

上述提到的 QueryRow、Exec 和 Qeury 只針對單一的 SQL Query，
那如果我們有多個 SQL Queries 的情況下，也就是 transaction，
就可以使用 tx 來確保 ACID。
舉例來說，如果今天有帳戶A需要轉帳給帳戶B時，這代表我們需要執行兩個 SQL Queries，
在這種情況下兩個 SQL Queries 必須要全部成功或全部失敗，也就是確保 Atomicity。
如果沒有將這些 SQL Queries 組成為一個 transaction，
就有可能發生帳戶A餘額被扣除，但帳戶B餘額沒有被增加的情況。

想多了解 transaction 和 ACID 可以參考 淺談關聯資料庫的 ACID

下圖的程式碼是使用轉帳的案例來示範，
可以看到我們會需要先使用 Begin，這代表我們啟動了一個 transaction，
再來就是使用 Rollback 確保當有錯誤發生時，已經成功的 SQL Queries 也必須 rollback，
至於其中 SQL Queries 則透過 Exec 來執行，
最後要執行的就是 Commit 代表生效所有的資料異動。

另外如果我們需要改變 transaction 的行為，
像是設定不同的 Isolation Level 就可以使用 BeignTx。

結論

一開始我們先分析了使用原生SQL或ORM的優缺點，
並且建議依據系統和場景來做出選擇。
接著在說明 pgx 的使用方式，像是 QueryRow、Exec 和 Query，
這三個都各自有不同的使用場景，並且也說明 transaction 的相關用法。
最後則是介紹 squirrel，這個套件能幫助我們在撰寫原生SQL上更加簡單及順手，並且也增加程式碼的可讀性。
